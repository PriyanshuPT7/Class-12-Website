Chapter 1: Database Management System

1. Data, Information & Features

Data are raw facts (e.g., numbers, text). Information is processed, organized data that carries meaning.

Features of good information: accurate, complete, timely, relevant, and understandable.

2. Database & Its Purpose

A database is a structured collection of related data stored to minimize redundancy and support efficient retrieval.

Purpose: reliable storage, integrity enforcement, multi-user access, backup/recovery, and easy querying.

3. Terminologies in Database

  • Table: Rows & columns storing related records.
  • Field/Column: Named attribute in a table.
  • Record/Row/Tuple: One complete set of field values.
  • Object: In OO-DBs, combines data with behavior.
  • Key: Attribute(s) that uniquely identify a record.

5. Data Dictionary

Repository of metadata: definitions of tables, columns, data types, relationships, constraints, usage and formats.

6. DBMS – Introduction, Objectives, Pros & Cons

Introduction

Software for defining, creating, querying, updating and administering databases (e.g., MySQL, Oracle).

Objectives

  • Efficient data storage & retrieval
  • Minimize redundancy
  • Ensure integrity & consistency
  • Support concurrent multi-user access
  • Backup & recovery

Advantages

  • Centralized control
  • Reduced duplication
  • Improved integrity
  • Powerful queries (SQL)
  • Security & auditing

Disadvantages

  • High setup cost
  • Complex administration
  • Requires trained personnel
  • Overhead for small datasets

7. Database Models

Hierarchical Model

Tree-like parent–child structure. Adv: simple navigation. Disadv: rigid, poor many-to-many support.

Network Model

Graph structure. Adv: flexible many-to-many. Disadv: complex schemas & queries.

Relational Model

Tables with rows & columns. Adv: intuitive, SQL support. Disadv: joins can be slow on large data.

Object-Relational Model

Extends relational with object features. Adv: complex data & inheritance. Disadv: less standardized support.

8. Integrity Constraints

  • Domain: Valid set/range for an attribute.
  • Entity: Unique identification (Primary Key).
  • Referential: Foreign key must match a primary key.
  • Key: Defines unique identifiers (Primary, Candidate, Alternate).
  1. Primary Key: A unique identifier for a record in a table.
  2. Foreign Key: A field in one table that uniquely identifies a row of another table.

9. Normalization

1NF

Ensures all columns contain atomic values and records are unique.

2NF

Removes partial dependencies; all non-key attributes depend on the whole primary key.

3NF

Removes transitive dependencies; non-key attributes are independent of each other.

Pros & Cons

  • Adv: reduces redundancy, improves integrity.
  • Disadv: more tables, complex joins.

11. Database Security

Challenges

  • Unauthorized access
  • SQL injection
  • Data leakage
  • Insider threats

Measures

  • Access control & authentication
  • Encryption at rest & in transit
  • Audit trails & monitoring
  • Regular patches & scans

12. DBA & Roles

DBA installs, configures, maintains the DBMS; designs schemas; tunes performance; enforces security; performs backups and recovery.

15. Constraints & Operators

  • NOT NULL: Disallow NULL values

Operators

16. WHERE & ORDER BY

SELECT Name, Age
FROM Students
WHERE Age > 18
ORDER BY Name DESC;

17. SQL JOINs

-- INNER JOIN
SELECT e.id, e.name, d.dept_name
FROM Employee e
INNER JOIN Department d
  ON e.dept_id = d.id;

-- LEFT JOIN
SELECT e.id, e.name, d.dept_name
FROM Employee e
LEFT JOIN Department d
  ON e.dept_id = d.id;

-- RIGHT JOIN
SELECT e.id, e.name, d.dept_name
FROM Employee e
RIGHT JOIN Department d
  ON e.dept_id = d.id;

-- FULL OUTER JOIN
SELECT e.id, e.name, d.dept_name
FROM Employee e
FULL OUTER JOIN Department d
  ON e.dept_id = d.id;
